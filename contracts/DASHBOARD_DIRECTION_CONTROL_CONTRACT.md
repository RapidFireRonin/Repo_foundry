# Dashboard Direction Control Contract v1

This contract defines the interactive dashboard surface Garrett uses to steer autonomous Repo Foundry agents without dropping into CLI commands or editing files by hand.

## Purpose

Make human direction a first-class runtime input. The dashboard must let Garrett submit, inspect, reprioritize, pause, and trace goals while agents continue to execute autonomously inside policy boundaries.

## Required input fields

The dashboard direction form must collect:

- `instruction`: the primary natural-language command or goal.
- `details`: optional supporting context, constraints, or acceptance criteria.
- `priority`: numeric priority where higher values are handled first.
- `source`: must be `dashboard` for dashboard-created directions.
- `created_at`: timestamp generated by the backend or trusted runtime.
- `status`: initial status, normally `queued`.

## Direction lifecycle

A direction moves through these states:

- `queued`: saved and available for agent selection.
- `claimed`: selected by an agent for the current cycle.
- `in_progress`: active implementation, review, or research work is underway.
- `blocked`: execution cannot safely continue without a policy change, credential, or missing dependency.
- `done`: completed with linked evidence.
- `paused`: intentionally ignored until reactivated.
- `superseded`: replaced by a newer or higher-priority direction.

## Backend requirements

The backend must provide an API for dashboard-created directions that:

- Validates non-empty instructions.
- Applies a safe default priority when none is supplied.
- Stores source, timestamp, status, priority, instruction, and details.
- Returns the created direction to the dashboard immediately.
- Writes an audit entry for every create, edit, pause, resume, reprioritize, and completion event.
- Rejects directions that request denied actions unless a policy flag explicitly allows them.

## Dashboard requirements

The dashboard must show:

- A prominent prompt-style input box labeled for directing agents.
- A priority selector or slider.
- A visible submit success or failure state.
- The current Human Direction Queue immediately after submission.
- Direction status, priority, timestamp, source, claimed agent, linked PRs/issues/runs, and latest note.
- Controls to pause, resume, reprioritize, or supersede a direction when policy allows it.

## Agent consumption requirements

Agents must read active directions before choosing work. They should prefer higher-priority dashboard directions unless blocked by policy, failed checks, or an active emergency such as broken CI.

Every cycle summary must mention:

- Top active dashboard direction.
- Claimed directions.
- Completed directions.
- Blocked directions and the exact blocker.
- Links to PRs, issues, commits, logs, or artifacts created from the direction.

## Audit requirements

Every dashboard direction event must include:

- Timestamp.
- Actor or source.
- Direction identifier.
- Previous and new status when changed.
- Previous and new priority when changed.
- Reason or note.
- Related artifact links when available.

## Safety requirements

Dashboard direction does not bypass the autonomy policy. Agents must still refuse or block denied, destructive, credential-changing, or out-of-workspace actions unless explicit policy permits them.

## Definition of done

This contract is implemented when Garrett can type a goal into the dashboard, submit it, see it appear in the Human Direction Queue, watch an agent claim it, and follow all resulting work through logs, PRs, cycle summaries, and audit entries.
